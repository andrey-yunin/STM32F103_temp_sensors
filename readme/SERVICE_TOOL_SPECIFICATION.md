# Спецификация ПО: Analyzer Service Studio (DDS-240 GUI)

## 1. Общие сведения
Настоящий документ описывает требования к разработке графического программного обеспечения (GUI) для настройки, калибровки и диагностики узлов биохимического анализатора, работающих по протоколу DDS-240 (CAN). ПО заменяет ранее планируемую CLI-оболочку для обеспечения более интуитивного взаимодействия с оборудованием.

## 2. Архитектура программного обеспечения
Система строится на базе языка **Python 3.10+** с использованием фреймворка **PySide6 (Qt for Python)**.

### 2.1. Уровни абстракции (Layered Design)
1.  **Transport Layer (CAN HAL):** Абстракция над физическим адаптером (библиотека `python-can`). Поддержка PCAN, SocketCAN, SLCAN.
2.  **Protocol Layer (DDS-240 Async Engine):** Асинхронное управление транзакциями (REQ-ACK-DATA-DONE). Реализация через `asyncio` для предотвращения фризов GUI.
3.  **Device Model Layer (Digital Twins):** Объекты-двойники физических устройств, хранящие текущее состояние (температуры, позиции, конфиг из Flash).
4.  **UI Layer (Qt View):** Реактивный интерфейс, отображающий состояние "цифровых двойников".

### 2.2. Тип приложения и позиционирование
Выбор архитектуры **Native Desktop Application (PySide6/Qt)** вместо веб-интерфейса обусловлен следующими критическими требованиями:
*   **Прямой доступ к железу (Direct Hardware Access):** Нативное ПО взаимодействует с драйверами CAN-адаптеров (PCAN, CANable) напрямую через системные вызовы, обеспечивая минимальные задержки (Latency), что невозможно в стандартном браузере.
*   **Реактивность реального времени (Real-time UI):** Обработка высокочастотного потока данных (до 200 фреймов/сек) при мониторинге шины требует высокой производительности графического движка для отрисовки графиков и индикации "Warm Finger" без задержек.
*   **Автономность (Standalone):** ПО должно работать "в поле" без необходимости развертывания локальных серверов или наличия интернет-соединения. Дистрибуция осуществляется в виде единого исполняемого файла (`.exe`).
*   **Надежность транзакций:** Использование `asyncio` в нативном приложении позволяет гарантированно соблюдать жесткие тайм-ауты протокола DDS-240, не блокируя пользовательский интерфейс.

### 2.3. Стратегия виртуальной разработки (Digital Twin)
Для ведения разработки в условиях отсутствия физического макета (платы и CAN-адаптера) применяется стратегия симуляции:
*   **Virtual CAN Bus:** Использование виртуальной шины (библиотека `python-can` интерфейс `virtual` или `vcan` в Linux).
*   **Executor Simulator:** Разработка Python-скрипта ("Цифровой двойник"), который полностью имитирует поведение STM32-исполнителя. Скрипт отвечает на сервисные запросы (0xFxxx), генерирует тестовые данные температур и возвращает ошибки (NACK) для проверки стрессоустойчивости GUI.
*   **Transparent Transition:** Код логики и интерфейса остается неизменным при переходе с виртуальной шины на реальный адаптер (CANable/PCAN) — меняется только строка конфигурации драйвера.

## 3. Концепция интерфейса (Modern GUI)
Вместо иерархических режимов доступа (Cisco-like) используется компонентно-ориентированный интерфейс.

### 3.1. Основные рабочие области (Views)
*   **Discovery Panel (Network Explorer):** Левая панель со списком всех обнаруженных устройств на шине. Группировка по типам (Thermo, Motion). Индикация статуса (Online/Offline).
*   **Central Workspace:** Контекстно-зависимая область, меняющаяся при выборе устройства.
*   **Diagnostic Console:** Нижняя панель для вывода системных сообщений, логов транзакций и ошибок (NACK).

### 3.2. Специализированные виджеты (Thermo Specific)
*   **"Warm Finger" Mapping Tool:** Графическая сетка датчиков. При физическом нагреве датчика соответствующая ячейка в GUI подсвечивается (Heat Map), позволяя инженеру перетащить (Drag & Drop) физический ROM ID в нужный логический канал.
*   **Live Trend View:** График изменения температуры в реальном времени для выбранных каналов.

## 4. Функциональные требования

### 4.1. Управление конфигурацией (Service Layer)
| Функция | Описание |
| :--- | :--- |
| **Node Scanner** | Автоматический перебор адресов 0x02-0x7F с использованием `SRV_GET_DEVICE_INFO`. |
| **Identity Verification** | Отображение 96-битного UID, типа устройства и версии прошивки. Сопоставление с локальной базой (Inventory). |
| **Remote Config** | Изменение NodeID, параметров калибровки и маппинга в RAM устройства. |
| **Flash Management** | Кнопки "Commit to Flash" (0xF003) и "Factory Reset" (0xF006) с подтверждением через Magic Keys. |
| **Batch Update** | Возможность прошивки/настройки группы однотипных плат. |

### 4.2. Транзакционная надежность
*   **Visual Handshake:** Индикация процесса обмена данными (прогресс-бары для многопакетных данных, например UID).
*   **Error Handling:** Всплывающие уведомления (Toasts) при получении NACK с расшифровкой кода ошибки (например, "Ошибка: Неверный защитный ключ").
*   **Automatic Retry:** Повторная отправка пакета при потере подтверждения (до 3-х попыток).

## 5. Инвентаризация и "Цифровой паспорт"
Система поддерживает файлы инвентаризации (`.json`), которые служат "золотым образом" системы.

**Логика сопоставления (The Identity Guard):**
При подключении к анализатору ПО сравнивает UID обнаруженных плат с файлом инвентаризации.
*   **Зеленый:** Все совпадает (UID + NodeID).
*   **Желтый:** UID совпадает, но NodeID изменился (предложение авто-ребинда).
*   **Красный:** На знакомом NodeID обнаружена чужая плата (UID mismatch). Блокировка операций до выяснения.

## 6. Технический стек и зависимости
*   **GUI:** PySide6 (Qt 6).
*   **CAN:** python-can + custom DDS-240 wrapper.
*   **Async:** `qasync` (интеграция asyncio в событийный цикл Qt).
*   **Serialization:** `construct` (для парсинга сложных структур данных).
*   **Data Storage:** SQLite (для хранения истории калибровок и логов).

## 8. Безопасность и Валидация
*   **CRC16 Validation:** Перед отправкой команды `FLASH_COMMIT` ПО обязано пересчитать CRC16 структуры настроек и передать её для проверки на стороне исполнителя.
*   **Critical Confirmations:** Операции `Reboot` и `Factory Reset` требуют двойного подтверждения от пользователя в GUI.
*   **Read-Only Mode:** Возможность запуска в режиме мониторинга без права изменения настроек.

## 9. Требования к среде разработки (Dev Environment)

Для обеспечения воспроизводимости сборки и качества кода устанавливаются следующие требования к окружению:

### 9.1. Программный стек
*   **ОС:** Windows 10/11 (для инженеров на выезде) или Linux (Ubuntu 22.04+) для разработки.
*   **Python:** Версия 3.10.x - 3.12.x.
*   **Virtual Env:** Обязательное использование `venv` или `poetry` для изоляции зависимостей.
*   **IDE:** VS Code (с плагинами Python, Qt) или PyCharm Professional.
*   **UI Design:** Qt Designer (входит в PySide6) для визуального редактирования `.ui` файлов.

### 9.2. Аппаратные интерфейсы (Hardware HAL)
Разработчик должен иметь доступ как минимум к одному из поддерживаемых адаптеров:
*   **Peak-System PCAN-USB ISO (IPEH-002022):** **Рекомендуемый стандарт** для промышленной эксплуатации и обслуживания "в поле". 
    *   *Обоснование:* Наличие гальванической развязки (до 500В) защищает управляющий ПК от земляных петель и пробоев силовой части анализатора (моторы, нагреватели). Обеспечивает максимальную стабильность при высоких электромагнитных помехах (EMI).
*   **CANable / CandleLight:** Допустимый бюджетный вариант для этапа разработки "на столе" (Dev-Stage). Требует осторожности при подключении к силовой шине из-за отсутствия изоляции в базовых версиях.
*   **Virtual CAN (vcan):** Для отладки логики транзакций в среде Linux (Ubuntu) без физического подключения к шине.

### 9.3. Линтинг и проверка качества (Quality Gates)
*   **Форматирование:** `black` (стандарт экосистемы).
*   **Проверка типов:** `mypy` (строгая типизация для `DDS-240 Engine`).
*   **Линтер:** `ruff` или `flake8`.
*   **Тестирование:** `pytest` для Unit-тестов логики транзакций и парсеров.

### 9.4. Сборка и дистрибуция
*   **PyInstaller / Nuitka:** Для компиляции в автономный `.exe` файл, не требующий установки Python на компьютер сервисного инженера.
*   **GitHub Actions:** Настроенный CI для автоматической сборки релизов при пуше тегов.
