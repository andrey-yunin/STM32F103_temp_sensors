# Руководство по тестированию: Имитация Исполнителя (PC as Executor)

Этот гид поможет вам проверить логику Дирижера (STM32H723), заставляя его "думать", что на шине есть реальные устройства (моторы, помпы, датчики).

---

## 1. Подготовка и структура ID
Адаптер должен быть настроен на скорость **1 Мбит/с**.
Все сообщения в системе используют **29-битный Extended ID**.

**Формула формирования ID:** `(Priority << 26) | (Type << 24) | (Dst << 16) | (Src << 8)`
*   **Dst Addr:** Для всех ваших ответов всегда `0x10` (Дирижер).
*   **Src Addr:** ID платы, которую вы имитируете (`0x20` - Моторы, `0x30` - Помпы, `0x40` - Термо).
*   **Msg Type:** `1` (ACK), `2` (NACK), `3` (DATA/DONE/LOG).

---

## 2. Анатомия команды: Как составлять пакеты самостоятельно

Разберем процесс сборки команды на примере: **"Запустить насос №2 на 5 секунд (5000 мс)"**.

### Шаг А: Сборка 29-битного ID
Формула: `(Priority << 26) | (Type << 24) | (Dst << 16) | (Src << 8)`

1.  **Priority**: `0` (Высокий для команд).
2.  **Type**: `0` (Тип сообщения: COMMAND).
3.  **Dst (Цель)**: `0x30` (Плата помп).
4.  **Src (Источник)**: `0x10` (Дирижер).

**Расчет в HEX:**
*   `(0x0 << 26)` = `0x00000000`
*   `(0x0 << 24)` = `0x00000000`
*   `(0x30 << 16)` = `0x00300000`
*   `(0x10 << 8)` = `0x00001000`
*   **ИТОГО ID:** `0x00301000`

### Шаг Б: Сборка Payload (8 байт данных)
Структура: `[0-1] Код`, `[2] Канал`, `[3-6] Параметр`, `[7] Резерв`. Данные упаковываются в **Little-Endian** (младший байт первым).

1.  **Код команды (0x0201)**: `01 02`
2.  **Индекс канала (2)**: `02`
3.  **Параметр (5000 мс)**:
    *   5000 в HEX = `0x00001388`.
    *   Разбиваем по байтам: `88 13 00 00`.
4.  **Резерв**: `00`.

**ИТОГО Payload:** `0102028813000000`

**Готовая команда для терминала:**
`cansend can0 00301000#0102028813000000`

Для `PUMP_RUN_DURATION` ответ `DONE` должен имитироваться только после истечения 5000 мс и выключения насоса. Быстрый `DONE` сразу после `ACK` для этой команды является ошибкой модели.

## 3. Сценарий: Имитация платы моторов (NodeID 0x20)

Когда Дирижер присылает команду `ROTATE` (код `0x0101`), вы должны отправить два сообщения:

### Шаг 1: Подтверждение приема (ACK)
Дирижер ждет его в течение 50 мс.
*   **Команда:** `cansend can0 05102000#0101000000000000`
*   **Разбор ID (`05102000`):** Prio=1, Type=1 (ACK), Dst=0x10, Src=0x20.
*   **Разбор Data:** `01 01` (Код 0x0101), `00 00` (Нет ошибок).

### Шаг 2: Завершение движения (DONE)
Отправляется, когда "мотор доехал".
*   **Команда:** `cansend can0 07102000#0101010000000000`
*   **Разбор ID (`07102000`):** Prio=1, Type=3 (DATA/DONE), Dst=0x10, Src=0x20.
*   **Разбор Data:** `01` (Тип DONE), `01 01` (Код команды 0x0101), `00` (Канал 0).

## 3. Сценарий: Имитация термодатчиков (NodeID 0x40)

Дирижер запрашивает температуру командой `GET_TEMP` (код `0x9011`).

### Шаг 1: Подтверждение (ACK)
*   **Команда:** `cansend can0 05104000#1190000000000000`
*   **Разбор Data:** `11 90` (Код 0x9011 в Little-Endian).

### Шаг 2: Передача температуры (DATA)
Отправим значение **37.5°C**. В протоколе это `375` (0x0177).
*   **Команда:** `cansend can0 07104000#0200770100000000`
*   **Разбор Data:**
    * `02` (Тип DATA).
    * `00` (Индекс датчика 0).
    * `77 01` (Значение 375 в Little-Endian).
    * `00 00 00 00` (Заполнение до 8 байт).

---

## 5. Сценарий: Приветствие Fluidics (Service Discovery)

При включении Дирижер ищет устройства командой `Get Info` (код `0xF001`). Если вы хотите, чтобы Дирижер "увидел" плату помп (ID 0x30):

*   **ACK:** `cansend can0 05103000#01F0000000000000`
*   **DATA 1:** `cansend can0 07103000#0280030101100000`
*   **DATA 2:** `cansend can0 07103000#0280000000000000`
*   **DATA 3:** `cansend can0 07103000#0280000000000000`
*   **DONE:** `cansend can0 07103000#0101F00000000000`
*   **Разбор DATA 1:**
    * `02` (Тип DATA).
    * `80` (EOT=1, Seq=0).
    * `03` (Тип устройства: Fluidic Controller).
    * `01 01` (Версия прошивки 1.1).
    * `10` (16 каналов: 13 насосов + 3 клапана).

---

## 6. Памятка по ошибкам (NACK)

Если вы хотите сымитировать ошибку (например, "Мотор занят"), отправьте NACK вместо ACK:
*   **Команда:** `cansend can0 06102000#0101030000000000`
*   **Разбор Data:** `01 01` (Команда), `03 00` (Код ошибки 0x0003 - Motor Busy).

---

## 7. Сценарий: ACK без DONE / имитация IWDG recovery

Этот сценарий нужен для проверки Дирижера без реальной платы. Он имитирует отказ Executor после приема команды: Дирижер получает `ACK`, но `DONE` не приходит. Такое поведение возможно при зависании доменной задачи и последующем IWDG reset.

Для Fluidics `PUMP_START 0`:

*   **ACK:** `cansend can0 05103000#0202000000000000`
*   **DONE не отправлять.**

Ожидаемая реакция Дирижера:

*   зафиксировать operation timeout ожидания `DONE`;
*   считать команду невыполненной или неподтвержденной;
*   не продолжать recipe step как успешный;
*   выполнить recovery/discovery узла после восстановления связи;
*   сверить `NodeID`, `device_type`, `UID` и только затем возвращать узел в сценарий.

## 8. Физический smoke-test Motion Executor (NodeID 0x20)

Этот сценарий фиксирует проверенный CAN-уровень реальной платы Motion без подключенных нагрузок. Цель теста - подтвердить транспорт, адресацию, strict `DLC=8`, порядок `ACK -> DATA -> DONE` и диагностический `GET_STATUS` до перехода к физическим нагрузкам.

### 8.1. Подготовка SocketCAN

```bash
# Настройка can0 выполняется на стороне стенда; скорость должна быть 1 Mbit/s.
# В первом терминале оставляем пассивный мониторинг шины.
candump -x -t a can0
```

Проверенные ID для пары Conductor `0x10` <-> Motion `0x20`:

| Направление | Тип | CAN ID |
|:------------|:----|:-------|
| Conductor -> Motion | COMMAND | `00201000` |
| Motion -> Conductor | ACK | `05102000` |
| Motion -> Conductor | NACK | `06102000` |
| Motion -> Conductor | DATA / DONE / LOG | `07102000` |

### 8.2. GET_DEVICE_INFO (0xF001)

```bash
# Запрашиваем паспорт Motion executor.
# Payload: 0xF001 little-endian, остальные байты заполнены нулями до DLC=8.
cansend can0 00201000#01F0000000000000
```

Ожидаемый и фактически подтвержденный результат: `ACK`, три DATA-кадра, затем `DONE`.

```text
TX 00201000 [8] 01 F0 00 00 00 00 00 00
RX 05102000 [8] 01 F0 00 00 00 00 00 00
RX 07102000 [8] 02 80 01 01 00 08 0C 22
RX 07102000 [8] 02 80 07 14 52 16 30 30
RX 07102000 [8] 02 80 30 30 30 32 00 00
RX 07102000 [8] 01 01 F0 00 00 00 00 00
```

Расшифровка первого DATA-кадра:

*   `device_type = 0x01` - Motion Controller.
*   `fw_major = 0x01`, `fw_minor = 0x00`.
*   `channels = 0x08` - 8 шаговых двигателей.

### 8.3. GET_UID (0xF004)

```bash
# Запрашиваем полный UID Motion executor.
# Ответ должен прийти DATA-кадрами до DONE.
cansend can0 00201000#04F0000000000000
```

Ожидаемый и фактически подтвержденный результат: `ACK`, два DATA-кадра, затем `DONE`.

```text
TX 00201000 [8] 04 F0 00 00 00 00 00 00
RX 05102000 [8] 04 F0 00 00 00 00 00 00
RX 07102000 [8] 02 80 0C 22 07 14 52 16
RX 07102000 [8] 02 80 30 30 30 30 30 32
RX 07102000 [8] 01 04 F0 00 00 00 00 00
```

Подтвержденный UID:

```text
0C 22 07 14 52 16 30 30 30 30 30 32
```

### 8.4. GET_STATUS и transport diagnostics

```bash
# Запрашиваем диагностический статус Motion executor.
# Ответ должен прийти DATA-метриками до DONE.
cansend can0 00201000#07F0000000000000
```

Ожидаемый и фактически подтвержденный результат: `ACK`, 18 DATA-метрик `0x0001..0x0012`, затем `DONE`.

Проверенные диагностические выводы:

| Тест | Команда | Ожидаемый и подтвержденный результат |
|:-----|:--------|:--------------------------------------|
| Short DLC counter | `cansend can0 00201000#01F0` | Ответ отсутствует, следующий `F007` показывает `DROP_WRONG_DLC (0x0009) = 1`. |
| Foreign destination counter | `cansend can0 00211000#01F0000000000000` | Ответ отсутствует, следующий `F007` показывает `DROP_WRONG_DST (0x0007) = 1`. |
| Wrong message type counter | `cansend can0 05201000#01F0000000000000` | Ответ отсутствует, следующий `F007` показывает `DROP_WRONG_TYPE (0x0008) = 1`. |
| Standard ID | `cansend can0 120#01F0000000000000` | Ответ отсутствует, `DROP_NOT_EXT (0x0006) = 0`, кадр отсекается аппаратным bxCAN-фильтром до firmware. |

Финальный подтвержденный снимок после negative tests:

```text
RX_TOTAL=5, TX_TOTAL=81,
DROP_NOT_EXT=0, DROP_WRONG_DST=1, DROP_WRONG_TYPE=1, DROP_WRONG_DLC=1,
queue overflow/HAL CAN error/CAN fault counters=0
```

CAN Level A/acceptance и `GET_STATUS` diagnostics подтверждены.

Подтверждены finite `MOTOR_ROTATE -> DONE`, `speed=0 -> NACK 0x0006`, `MOTOR_BUSY` внутри `TIM1/TIM2 group`, параллельные движения в разных TIM-группах, `START_CONTINUOUS speed>0 + STOP`, а также `F002/F003/F005/F006`.
