# ТЗ: исполнитель датчиков положения и юстировки

Дата: 07.05.2026  
Статус: черновик для обсуждения  
Область: новая плата-исполнитель на STM32F103 для датчиков, связанных с позиционированием и юстировкой механизмов DDS-240.

## 1. Назначение документа

Документ формирует начальное техническое задание на отдельный CAN-исполнитель датчиков.

В экосистеме предполагаются два датчиковых исполнителя:

1. **Thermo Executor** - исполнитель термодатчиков. Уже разработан, но может потребовать отдельного рефакторинга под общий стандарт.
2. **Position/Sensor Executor** - новый исполнитель для датчиков положения, юстировки и других нетемпературных датчиков.

На текущем этапе рассматривается только часть, связанная с позиционированием и юстировкой. Датчики уровня жидкостей и температурный контур в этот блок пока не включаются.

## 2. Краткая картина работы

Sensor Executor - это нижний уровень сбора данных с датчиков положения. Он не управляет моторами, насосами или клапанами и не принимает системных решений.

Типовой сценарий:

```text
Power-on
-> Load Flash config
-> CAN discovery by Conductor
-> Sensor polling GPIO/ADC
-> Conductor requests 0x9020/0x9021/0x90F0
-> Sensor Executor returns normalized state
-> Conductor decides allow/block/adjust
```

В режиме юстировки Sensor Executor измеряет профиль датчика, а Дирижер рассчитывает offset и принимает решение:

```text
Hall ADC profile
-> reference point
-> drift
-> Conductor decision
-> keep offset / auto-adjust offset / stop process
```

## 3. Поток данных

Штатный поток состояния датчика:

```text
GPIO/ADC
-> sensor_driver
-> sensor_device_mapping
-> task_sensor_controller
-> task_dispatcher
-> CAN
-> Дирижер
```

Поток юстировки реакционного диска:

```text
Hall digital/ADC signal
-> filtered profile
-> reference_step
-> drift_steps
-> Conductor tolerance check
-> home_offset_steps update or fault
```

Sensor Executor выполняет низкоуровневое измерение и фильтрацию. Дирижер остается владельцем системного состояния и решений.

Для текущей архитектуры принято решение: юстировка роторов выполняется в сервисном режиме медленного вращения. Дирижер дает Motion малый шаг или малый пакет шагов, дожидается полной остановки и только после этого читает Sensor Executor. Такой режим не требует realtime-захвата фронта через CAN.

## 4. Минимальный CAN-контракт Sensor Executor

Минимально новая плата должна поддерживать:

| Команда | Назначение |
|---------|------------|
| `0xF001 GET_DEVICE_INFO` | Discovery и проверка типа устройства |
| `0xF007 GET_STATUS` | Диагностика, fault flags, transport/domain metrics |
| `0x9020 SENSOR_GET_ALL_POSITIONS` | Получить все датчики положения |
| `0x9021 SENSOR_GET_POSITION` | Получить один датчик положения |
| `0x90F0 SENSOR_LIST` | Получить список датчиков и флаги `active/calibrated` |

Команды конфигурации и расширенные сервисные команды юстировки требуют отдельного уточнения формата.

## 5. Состояния датчиков и ошибки

Базовые состояния логического датчика:

| Состояние | Смысл | Действие Дирижера |
|-----------|-------|-------------------|
| `inactive` | Датчик есть в таблице, но отключен конфигурацией | Не использовать в технологической логике |
| `active` | Датчик включен и опрашивается | Можно читать состояние |
| `calibrated` | Датчик разрешен для технологических сценариев | Можно использовать в проверках и юстировке |
| `fault` | Ошибка датчика или канала | Блокировать зависимые операции |
| `stale` | Данные устарели | Повторить опрос или перевести ресурс в degraded |
| `degraded` | Датчик работает, но качество ниже нормы | Разрешение зависит от сценария и допуска |

Поддерживаемые смысловые классы датчиков:

| Класс | Назначение |
|-------|------------|
| `home_sensor` | Поиск или подтверждение исходной позиции |
| `limit_sensor` | Контроль крайнего положения |
| `cover_sensor` | Контроль крышек |
| `position_code_sensor` | Определение рабочей позиции по кодовым прорезям/меткам |
| `profile_sensor` | Снятие ADC-профиля для offset/юстировки |

Связанные ошибки Host API:

- `ERR_SENS_POS_GENERAL`;
- `ERR_SENS_POS_HOME`;
- `ERR_SENS_COVER_OPEN`;
- `ERR_SENS_COVER_ERROR`.

## 7. Архитектурная роль нового исполнителя

Новый исполнитель не должен быть "исполнителем юстировки" в узком смысле. Его базовая роль - надежно читать физические датчики и отдавать Дирижеру нормализованное состояние.

Юстировка и калибровка позиционирования выполняются не внутри датчиковой платы как самостоятельный технологический процесс, а через связку:

- датчики фиксируют физические признаки положения;
- Дирижер хранит системное состояние и принимает решения;
- Motion/Fluidics и другие исполнители выполняют физические действия;
- Дирижер проверяет, что действие допустимо по состоянию ресурсов и датчиков.

Такой подход сохраняет централизованное управление в Дирижере и не размазывает системную логику по нескольким платам.

## 8. Разделение ответственности

### 8.1. Host

Host работает с высокоуровневыми командами `0x90xx`.

По текущей документации Host умеет:

- запросить состояние всех датчиков положения;
- запросить один датчик положения;
- запросить крышки;
- получить список датчиков;
- читать/записывать настройки датчиков.

Отдельной Host-команды "запустить юстировку позиционирования" в текущем разделе `0x90xx` нет. Значит юстировка на первом этапе рассматривается как внутренняя работа пары Дирижер - Executor, а не как публичная команда Host.

### 8.2. Дирижер

Дирижер является владельцем системного состояния.

Его задачи:

- опрашивать датчиковый исполнитель;
- хранить актуальный кэш состояний датчиков;
- проверять свежесть данных;
- запрещать опасные или невозможные действия;
- связывать датчики с ресурсами и механизмами;
- выполнять процедуры поиска исходного положения и юстировки через команды к другим исполнителям;
- отдавать Host ответы по `0x90xx`.

Этот слой можно рассматривать как будущий **Resource Guard / Process Guard**: проверяется не только готовность мотора, насоса или клапана, но и состояние обслуживаемого ресурса.

### 8.3. Sensor Executor

Новый исполнитель отвечает за нижний уровень датчиков.

Его задачи:

- читать входы STM32F103;
- поддерживать датчики Холла, оптодатчики и другие дискретные датчики положения;
- выполнять антидребезг и фильтрацию;
- учитывать полярность входа;
- хранить карту `физический вход -> логический sensor_id`;
- отдавать логическое состояние датчиков по CAN;
- хранить флаг активности и флаг калибровки;
- отдавать диагностическое состояние платы.

Исполнитель не должен сам принимать системные решения уровня "можно запускать анализ" или "можно включать насос". Эти решения остаются в Дирижере.

### 8.4. Motion/Fluidics Executor

Motion, Fluidics и другие исполнители остаются исполнителями физических действий.

Если сигнал датчика является жесткой защитой и требует реакции быстрее CAN-цикла, он должен обрабатываться локально аппаратно или на том исполнителе, который управляет опасной нагрузкой.

Если сигнал нужен для проверки положения, юстировки, готовности ресурса или штатного технологического контроля, его можно вести через Sensor Executor и централизованную логику Дирижера.

## 9. Первичный набор датчиков положения

Текущий Host API уже задает логические идентификаторы:

| ID | Имя | Назначение |
|----|-----|------------|
| `0x01` | `POS_REACTION_HOME` | Исходная позиция реакционного диска |
| `0x02` | `POS_REAGENT_HOME` | Исходная позиция ротора реагентов |
| `0x03` | `POS_SAMPLE_HOME` | Исходная позиция диска образцов |
| `0x10` | `POS_DISPENSER_Z` | Датчик Z дозатора |
| `0x11` | `POS_DISPENSER_X` | Датчик X дозатора |
| `0x20` | `POS_MIXER_Z` | Датчик Z миксера |
| `0x30` | `COVER_MAIN` | Основная крышка |
| `0x31` | `COVER_REAGENT` | Крышка реагентов |
| `0x32` | `COVER_SAMPLE` | Крышка образцов |

Крышки технически находятся в разделе датчиков положения, но для юстировки они являются скорее блокировочными/охранными ресурсами. Их нужно учитывать отдельно от датчиков home/позиции.

## 10. Кодовые датчики положения

Некоторые механизмы могут иметь не одну метку `HOME`, а несколько рабочих меток с разным физическим рисунком. Пример: дозатор с прорезями для положений `CUVETTE`, `REAGENT`, `SAMPLE`, `WASH`.

Если прорези отличаются шириной, расстоянием или последовательностью фронтов, такой оптодатчик работает как простой позиционный кодер.

### 10.1. Что измеряется

Для одного прохода через кодовую метку нужно фиксировать:

```text
rising_edge_step
falling_edge_step
width_steps
gap_from_prev_steps
digital_state
quality
```

Если используется ADC/аналоговая оптика, дополнительно можно хранить уровень сигнала и качество фронта.

### 10.2. Карта позиций

Дирижер должен иметь таблицу соответствия:

```text
position_id -> expected_width_steps / expected_gap_steps / tolerance
```

Пример:

| Позиция | Смысл |
|---------|-------|
| `DISPENSER_POS_CUVETTE` | Дозатор над кюветой |
| `DISPENSER_POS_REAGENT` | Дозатор над реагентом |
| `DISPENSER_POS_SAMPLE` | Дозатор над образцом |
| `DISPENSER_POS_WASH` | Дозатор над промывочной станцией |

Финальные имена и коды позиций должны быть согласованы с механикой конкретного узла.

### 10.3. Использование при инициализации

При старте системы кодовый датчик может использоваться для построения карты текущего положения механизма:

1. Дирижер запускает сервисное сканирование.
2. Motion выполняет медленное движение малыми шагами/пакетами.
3. Sensor Executor считывает состояние оптодатчика после остановки.
4. Дирижер фиксирует фронты и ширину метки.
5. По таблице кодов определяется текущая рабочая позиция.
6. Если код распознан - механизм считается позиционно подтвержденным.
7. Если код не распознан - фиксируется ошибка позиционирования/юстировки.

Такой датчик не заменяет проверку механики, но позволяет:

- определить текущую позицию после включения;
- подтвердить достижение рабочей точки;
- обнаружить потерю шагов;
- выявить смещение, люфт или ошибку юстировки;
- не полагаться только на слепой счет шагов от последнего `HOME`.

### 10.4. Граница ответственности

Sensor Executor возвращает измеренное состояние или подготовленный результат по метке. Дирижер сопоставляет результат с картой позиций и принимает системное решение.

Для текущей архитектуры кодовые датчики используются в сервисном медленном режиме. Точный realtime-кодер движения не реализуется через CAN.

## 11. Команды Host и внутренняя маршрутизация

### 11.1. `0x9020 SENSOR_GET_ALL_POSITIONS`

Host запрашивает состояние всех датчиков положения.

Вариант реализации в Дирижере:

- если кэш свежий, ответить из кэша;
- если кэш устарел, запросить Sensor Executor;
- если исполнитель недоступен, вернуть ошибку датчикового контура.

### 11.2. `0x9021 SENSOR_GET_POSITION`

Host запрашивает один датчик положения.

Дирижер должен вернуть нормализованное состояние `0/1`, а не сырой уровень GPIO.

### 11.3. `0x9040 SENSOR_GET_COVERS`

Крышки можно физически обслуживать тем же Sensor Executor, но логически это отдельная группа условий безопасности.

### 11.4. `0x9050 SENSOR_CONFIG` и `0x9060 SENSOR_GET_CONFIG`

Текущая документация описывает параметры для температуры и жидкостей, но не раскрывает параметры для датчиков положения.

Для датчиков положения требуется дополнительно определить:

- полярность входа;
- время антидребезга;
- активность датчика;
- привязку логического `sensor_id` к физическому входу;
- признак "калиброван";
- возможно, режим датчика: home, limit, cover, service.

Пока не решено, эти параметры будут открыты Host через `0x9050/0x9060` или останутся сервисной настройкой Дирижер - Executor.

### 11.5. `0x90F0 SENSOR_LIST`

Команда важна для диагностики.

Флаг `active` показывает, что датчик включен в конфигурацию.  
Флаг `calibrated` показывает, что датчик прошел процедуру привязки/проверки и может использоваться в технологических сценариях.

## 12. Предлагаемая внутренняя модель Sensor Executor

Минимальная модель одного логического датчика:

| Поле | Назначение |
|------|------------|
| `sensor_type` | Тип датчика: положение, крышка, резерв |
| `sensor_id` | Логический ID из Host API |
| `physical_channel` | Номер входа STM32F103 |
| `polarity` | Нормально открытый/нормально закрытый или инверсия |
| `debounce_ms` | Время подтверждения изменения состояния |
| `state` | Нормализованное состояние |
| `raw_state` | Сырой уровень входа для сервиса |
| `active` | Датчик используется |
| `calibrated` | Датчик разрешен для технологической логики |
| `fault` | Признак ошибки датчика или канала |

Эта модель не должна зависеть от конкретного механизма. Например, датчик `POS_MIXER_Z` остается датчиком положения, а не частью "модуля миксера".

## 13. Юстировка и калибровка позиционирования

Здесь важно не смешивать два разных понятия.

В Дирижере уже есть трансляторы/калибраторы, которые переводят параметры Host в физические величины: объем в длительность насоса, номер кюветы в шаги мотора и т.д.

Новый раздел относится к физической юстировке:

- поиск исходного положения;
- проверка, что датчик home реально срабатывает;
- сохранение признака "узел откалиброван";
- проверка допустимости движения перед выполнением рецепта;
- диагностика несоответствия: команда выполнена, но ожидаемый датчик не подтвердил состояние.

## 14. Важный архитектурный риск

Если датчик home физически подключен только к Sensor Executor, а мотором управляет Motion Executor, то остановка мотора по датчику через Дирижер и CAN не является жестким realtime-контуром.

Для текущей архитектуры это принято как рабочая схема: юстировка выполняется не на рабочей скорости, а в специальном сервисном режиме медленного вращения.

Принцип:

```text
Motion: сделать N шагов и остановиться
Sensor Executor: измерить состояние/ADC
Дирижер: принять решение
```

Так задержки CAN/RTOS не искажают точку измерения, потому что датчик читается после остановки.

Для быстрого homing или защиты от механического удара нужен один из вариантов:

- датчик home подключается напрямую к Motion Executor;
- сигнал дублируется на Motion и Sensor Executor;
- используется аппаратная защита;
- движение в режиме юстировки выполняется малыми шагами под контролем Дирижера.

Это решение нужно принять до проектирования платы и назначения GPIO.

## 15. Пример процесса юстировки: реакционный диск

Раздел фиксирует согласованную рабочую логику на примере `POS_REACTION_HOME`.

### 15.1. Первичная механическая установка

Первый шаг - грубая механическая установка.

1. Реакционный диск устанавливается в реальный механический `HOME`.
2. Магнит датчика Холла выставляется напротив датчика в зоне этого положения.
3. Положение магнита вручную корректируется после закрепления.
4. Магнит фиксируется.
5. Выполняется несколько медленных проходов через зону датчика для проверки повторяемости.

Эта операция не задает финальный программный ноль. Она только создает физическую реперную зону рядом с правильным `HOME`.

### 15.2. Дискретный и аналоговый сигнал Холла

Для датчика Холла нужно учитывать два уровня данных:

- цифровой сигнал `0/1` - грубое обнаружение зоны `HOME`;
- аналоговый сигнал через ADC - профиль магнитного поля для точной юстировки.

Sensor Executor должен уметь читать и нормализовать цифровой вход, а также снимать ADC-профиль при медленном проходе через магнит.

Минимальные данные для сервисной юстировки:

| Данные | Назначение |
|--------|------------|
| `raw_adc` | Сырое значение ADC |
| `filtered_adc` | Отфильтрованное значение ADC |
| `adc_min/adc_max` | Границы профиля |
| `reference_step` | Шаг, на котором найдена характерная точка |
| `reference_type` | Пик, центр окна или фронт по порогу |

### 15.3. Расчет `home_offset_steps`

При медленном проходе реакционного диска через магнит формируется ADC-профиль. Значение ADC растет, достигает характерной точки, затем падает.

Характерная точка профиля становится датчиковой реперной точкой. Это может быть:

- пик ADC;
- центр окна между двумя пороговыми фронтами;
- выбранный фронт по порогу.

Если реальный механический `HOME` не совпадает с этой точкой, вводится смещение:

```text
home_offset_steps = mechanical_home_steps - reference_step
```

Пример:

```text
reference_step = 12000
mechanical_home_steps = 12035
home_offset_steps = +35
```

При штатном поиске `HOME` Дирижер находит датчиковую реперную точку, добавляет `home_offset_steps` и считает полученную позицию логическим нулем реакционного диска.

### 15.4. Автоподстройка offset

Автоюстировка - это контролируемое изменение `home_offset_steps` по результатам повторного измерения ADC-профиля.

Базовая логика:

```text
drift_steps = current_reference_steps - saved_reference_steps
new_home_offset_steps = old_home_offset_steps - drift_steps
```

Пример:

```text
saved_reference_steps = 12000
old_home_offset_steps = +35
current_reference_steps = 12008
drift_steps = +8
new_home_offset_steps = +27
```

Такой пересчет удерживает логический `HOME` в прежней механической точке, если датчиковая реперная точка немного сместилась.

### 15.5. Ограничения автоюстировки

Автоподстройка не должна маскировать серьезную механическую проблему.

Требования:

- не сохранять коррекцию от одного случайного измерения;
- подтверждать дрейф несколькими проходами;
- игнорировать микродрейф в пределах шумового допуска;
- разрешать изменение offset только в заданном диапазоне;
- при превышении допуска выдавать ошибку юстировки или механики;
- хранить последний drift и счетчик коррекций.

Если смещение стало большим, это не штатная автоюстировка, а признак проблемы: магнит ослаб, датчик сместился, появился люфт или изменилась механика диска.

### 15.6. Разделение ролей при юстировке

| Участник | Роль |
|----------|------|
| Sensor Executor | Читает цифровой Холл, снимает ADC-профиль, фильтрует данные |
| Motion Executor | Выполняет медленное движение реакционного диска |
| Дирижер | Управляет процедурой, считает offset, проверяет допуски, сохраняет состояние |
| Host | На текущем этапе только запрашивает состояние датчиков через `0x90xx` |

### 15.7. Нагрузка на Дирижер и режимы обмена

Дирижер не должен получать непрерывный поток ADC-точек в штатном режиме.

Низкоуровневая обработка выполняется на Sensor Executor:

- частое чтение ADC;
- фильтрация профиля;
- поиск пика, центра окна или фронта;
- оценка качества сигнала;
- хранение последнего результата измерения.

Дирижер получает компактный результат и принимает системное решение:

```text
sensor_id = POS_REACTION_HOME
reference_step_delta = +8
peak_adc = 3120
quality = OK
status = AUTO_ADJUST_CANDIDATE
```

Рекомендуемые режимы:

| Режим | Назначение |
|-------|------------|
| Normal | Sensor Executor локально обрабатывает профиль и отдает только итоговое событие/результат |
| Service | Дирижер запрашивает расширенные данные для сервисной юстировки |
| Debug | Снятие массива ADC-точек вручную для диагностики, не используется в рабочем цикле |

Так Дирижер остается владельцем системных решений, но не становится обработчиком ADC.

### 15.8. Рабочая логика контроля offset

При инициализации Дирижер поднимает из своей Flash:

- `home_offset_steps`;
- `saved_reference_steps`;
- допуски автоюстировки;
- флаг `calibrated`.

Sensor Executor хранит конфигурацию датчика: канал, полярность, фильтры, активность и ADC-параметры. Он не хранит системную истину по механике диска.

Рабочий цикл:

1. Дирижер переводит систему в сервисный режим юстировки.
2. Motion выполняет малый шаг или малый пакет шагов и останавливается.
3. Дирижер запрашивает Sensor Executor.
4. Sensor Executor измеряет состояние/ADC и формирует результат.
5. Дирижер повторяет цикл до построения профиля.
6. Дирижер рассчитывает `drift_steps`.
7. Дирижер принимает решение по допускам.

Правило принятия решения:

```text
abs(drift) <= deadband
    -> ничего не менять

deadband < abs(drift) <= auto_adjust_limit
    -> применить коррекцию offset

abs(drift) > auto_adjust_limit
    -> остановить или заблокировать процесс
```

Коррекцию во Flash нельзя сохранять по одному случайному измерению. Дрейф должен быть подтвержден несколькими проходами. Допустимо сначала применить коррекцию в RAM, а во Flash сохранить только стабильное значение.

### 15.9. Рекомендуемые параметры сервисного сканирования

Скорость юстировки должна быть отдельной конфигурацией, не равной штатной скорости вращения ротора в рабочем рецепте.

Для текущей архитектуры точность задается не только скоростью, а полным циклом:

```text
малый шаг/пакет шагов
-> полная остановка
-> задержка стабилизации
-> чтение digital/ADC
```

Начальные значения для стендовой отладки:

| Фаза | Скорость Motion | Шаг сканирования | Назначение |
|------|-----------------|------------------|------------|
| Coarse search | `200..500 steps/s` | `20..50 steps` | Быстро найти зону датчика |
| Backoff | `100..300 steps/s` | до выхода из зоны | Вернуться перед точным проходом |
| Fine scan | `50..100 steps/s` | `1..5 steps` | Найти фронт/центр/пик |
| Verification | `50..100 steps/s` | `1..5 steps` | Повторить проход и проверить drift |

После каждой остановки требуется выдержка перед измерением:

```text
settle_delay_ms = 10..30 ms
```

Эти значения являются стартовыми для наладки. Финальные значения должны быть подобраны по механике конкретного ротора: редуктор, люфт, масса диска, ширина зоны датчика, требуемая точность offset.

Критерий качества сканирования: активная зона датчика должна давать достаточно точек измерения. Для точной юстировки желательно получать не менее `10..20` точек на ширину зоны срабатывания.

## 16. Начальные требования к CAN-интеграции

Новый исполнитель должен соответствовать общей экосистеме:

- CAN 1 Mbit/s;
- 29-bit Extended ID;
- команды от Дирижера с `DLC=8`;
- Little-Endian payload;
- поддержка broadcast discovery;
- поддержка service-команд `0xF001..0xF007` по общему стандарту;
- счетчики диагностики транспорта;
- safe-state при ошибках.

Текущие NodeID заняты:

| NodeID | Узел |
|--------|------|
| `0x10` | Conductor |
| `0x20` | Motion |
| `0x30` | Fluidics |
| `0x40` | Thermo |

NodeID нового Sensor Executor пока не назначен. Рабочий кандидат для обсуждения: `0x50`.

Device Type нового исполнителя также не назначен. Рабочий кандидат для обсуждения: `0x04`.

## 17. RTOS и файловая структура Sensor Executor

Sensor Executor должен строиться по единому каркасу Executor-плат DDS-240, без введения новой архитектурной схемы.

### 17.1. RTOS-задачи

Базовый набор задач:

| Задача | Назначение |
|--------|------------|
| `task_can_handler` | Прием/передача CAN, transport-фильтрация, очереди RX/TX |
| `task_dispatcher` | Разбор команд, service layer, ACK/NACK/DATA/DONE, маршрутизация в домен |
| `task_sensor_controller` | Опрос GPIO/ADC, фильтрация, состояние датчиков, подготовка результатов |
| `task_watchdog` | Supervisor heartbeat и единственное место refresh IWDG |

Дополнительная задача для ADC/DMA не является базовой частью v1. Она допускается позже только при подтвержденной необходимости плотного ADC-профиля в service/debug режиме.

### 17.2. Обычные модули

Обычные `.c/.h` файлы не являются RTOS-задачами. Они вызываются из задач.

| Модуль | Назначение |
|--------|------------|
| `sensor_driver` | Низкий уровень GPIO/ADC, raw/filtered значения |
| `sensor_device_mapping` | Связь логического `sensor_id` с физическим входом, полярностью и ADC-каналом |
| `app_flash` | Хранение конфигурации Sensor Executor во Flash |
| `app_watchdog` | Heartbeat API для критических задач |
| `app_config` | Константы платы, размеры очередей, timeout, NodeID/DeviceType |
| `app_queues` | Очереди и handles RTOS-объектов |
| `can_protocol` / `can_transport` | CAN ID, payload, ACK/NACK/DATA/DONE |
| `diagnostics` / `status` | Метрики, fault flags, snapshot для `GET_STATUS` |
| `safe_state` | Перевод Sensor Executor в безопасное состояние |

### 17.3. Разделение ответственности внутри Sensor Executor

`sensor_driver` не должен знать технологический смысл датчика. Его уровень - прочитать GPIO/ADC и вернуть сырые или отфильтрованные значения.

`sensor_device_mapping` связывает физические каналы с логическими датчиками:

```text
physical_channel -> sensor_id
gpio_polarity
adc_channel
debounce_ms
active
calibrated
```

`task_sensor_controller` работает с логическими датчиками: состояние `active/calibrated/fault`, профиль Холла, результат юстировки, готовность данных для Дирижера.

## 18. Минимальный набор функций v1

Для первого этапа разработки Sensor Executor достаточно:

- поднять CAN-транспорт по общему стандарту;
- реализовать `GET_INFO`, `GET_STATUS`, `REBOOT`, `COMMIT`, `FACTORY`;
- описать таблицу логических датчиков положения;
- читать дискретные входы;
- нормализовать состояние входов;
- реализовать антидребезг;
- вернуть один датчик положения;
- вернуть все датчики положения;
- вернуть список датчиков с флагами `active/calibrated`;
- сохранить конфигурацию во Flash;
- выдать ошибку, если датчик неактивен, не откалиброван или неисправен.

## 19. Flash-конфигурация и владение данными

Важно разделить, какие данные принадлежат Sensor Executor, а какие Дирижеру.

### 19.1. Sensor Executor Flash

Sensor Executor хранит только конфигурацию своей платы и датчиков:

```text
config_magic
config_version
crc
node_id
device_type
sensor_count
sensor_table[]
```

Минимальная запись `sensor_table`:

```text
sensor_id
sensor_class
physical_channel
gpio_polarity
adc_channel
debounce_ms
active
calibrated
```

Sensor Executor не хранит системные offset механизмов как главный источник истины.

### 19.2. Flash/конфигурация Дирижера

Дирижер хранит системные параметры позиционирования:

```text
home_offset_steps
saved_reference_steps
deadband_steps
auto_adjust_limit_steps
position_code_map
sensor_to_mechanism_map
last_drift_steps
calibration_state
```

Именно Дирижер решает, можно ли применять offset, обновлять его или блокировать сценарий.

## 20. Диагностика и GET_STATUS

Sensor Executor должен отдавать общие transport/service метрики по стандарту экосистемы и доменные датчиковые метрики.

Минимальные доменные метрики:

| Метрика | Назначение |
|---------|------------|
| `sensor_poll_count` | Количество циклов опроса датчиков |
| `gpio_fault_count` | Ошибки дискретных входов |
| `adc_fault_count` | Ошибки ADC |
| `stale_count` | Количество устаревших значений |
| `profile_quality_fault_count` | Плохое качество профиля при юстировке |
| `uncalibrated_access_count` | Попытка использовать неоткалиброванный датчик |
| `config_crc_error_count` | Ошибки CRC конфигурации |

Формат metric_id для этих значений нужно назначить после согласования Sensor Executor в общей экосистеме.

## 21. Критерии приемки v1

Первый этап Sensor Executor считается готовым, если выполнены проверки:

1. `0xF001 GET_DEVICE_INFO` возвращает ожидаемый NodeID/DeviceType/количество каналов.
2. `0xF007 GET_STATUS` возвращает transport/service metrics без ошибок после старта.
3. `0x9021 SENSOR_GET_POSITION` возвращает состояние одного датчика.
4. `0x9020 SENSOR_GET_ALL_POSITIONS` возвращает список всех активных датчиков положения.
5. `0x90F0 SENSOR_LIST` возвращает список датчиков с флагами `active/calibrated`.
6. Некорректный `sensor_id` дает управляемую ошибку, а не зависание.
7. `inactive/fault/stale/uncalibrated` корректно различаются.
8. Flash-конфигурация читается при старте и проверяется по CRC.
9. Ошибка CRC переводит плату в безопасное состояние конфигурации.
10. Сервисное медленное сканирование позволяет построить профиль датчика или кодовой метки.
11. Дирижер может заблокировать зависимую операцию по `fault/stale/uncalibrated`.
12. Watchdog не срабатывает в простое и срабатывает при зависании критической задачи.

## 22. Граница v1/v2

### 22.1. Входит в v1

- дискретные датчики положения;
- базовое чтение ADC для profile-сенсоров;
- `active/calibrated/fault/stale`;
- Flash-конфигурация Sensor Executor;
- `GET_INFO`, `GET_STATUS`, `GET_POSITION`, `GET_ALL_POSITIONS`, `SENSOR_LIST`;
- сервисное медленное сканирование через Дирижер;
- базовая поддержка кодовых меток как результата сервисного сканирования.

### 22.2. Кандидаты v2

- расширенные DATA-ответы с профилем;
- событийные уведомления при изменении датчика;
- ADC DMA/service debug mode;
- автоматическое распознавание кодовых меток внутри Sensor Executor;
- расширенная диагностика качества фронтов;
- Host-команда запуска юстировки, если она будет утверждена.

### 22.3. Не входит в текущую архитектуру

- realtime-захват фронта через CAN во время быстрого движения;
- прямое управление Motion/Fluidics со стороны Sensor Executor;
- перенос системных offset из Дирижера в Sensor Executor как главный источник истины.

## 23. Что не входит в текущий этап

На текущем этапе не включаем:

- температурные датчики;
- жидкостные датчики;
- алгоритмы расчета уровней жидкостей;
- управление насосами, клапанами или моторами с Sensor Executor;
- полноценную Host-команду запуска юстировки;
- изменение общей документации экосистемы без отдельного решения.

## 24. Открытые вопросы

Перед переходом к проектированию кода нужно согласовать:

1. Окончательный NodeID нового исполнителя.
2. Окончательный Device Type.
3. Количество физических входов STM32F103.
4. Какие датчики подключаются только к Sensor Executor, а какие должны дублироваться на Motion/Fluidics.
5. Нужно ли Sensor Executor отправлять события по изменению состояния, или на первом этапе достаточно опроса Дирижером.
6. Какой timeout свежести датчикового кэша нужен в Дирижере.
7. Какие параметры датчиков положения открываем Host через `0x9050/0x9060`.
8. Какая процедура переводит датчик в состояние `calibrated`.
9. Должна ли юстировка быть полностью внутренней процедурой Дирижера или позже появится отдельная Host-команда.
10. Какие ADC-данные Sensor Executor должен отдавать Дирижеру при сервисной юстировке.
11. Где хранится `home_offset_steps`: в Дирижере, в Sensor Executor или в общей конфигурации.
12. Какие предельные значения drift считать допустимой автоподстройкой, а какие - ошибкой механики.
13. Какие механизмы имеют кодовые датчики положения и какие таблицы кодов должны храниться в Дирижере.
14. Какие domain metric_id назначаются для Sensor Executor в `GET_STATUS`.

## 25. Предварительное решение

На текущий момент рабочая модель такая:

- Sensor Executor является нижним уровнем сбора датчиков положения.
- Дирижер является центральным владельцем состояния и Resource Guard.
- Host получает состояние через существующие команды `0x90xx`.
- Юстировка позиционирования пока считается внутренней процедурой Дирижера.
- Быстрые аварийные реакции не должны зависеть только от CAN-опроса.

Документ будет уточняться после согласования открытых вопросов.
