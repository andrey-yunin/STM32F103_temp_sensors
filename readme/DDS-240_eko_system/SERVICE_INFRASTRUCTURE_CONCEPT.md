# Концепция сервисной инфраструктуры и унифицированного протокола (DDS-240)

## 1. Введение
Настоящий документ описывает архитектурный подход к созданию сервисного программного обеспечения для обслуживания биохимического анализатора. Основная цель — обеспечить единый интерфейс доступа ко всем узлам системы (исполнителям) через общую шину CAN, используя стандартизированный набор сервисных команд.

## 2. Уровни архитектуры сервисного ПО (PC Side)

Для обеспечения масштабируемости и надежности, ПО верхнего уровня (Service Tool) должно быть разделено на следующие слои:

### 2.1. Транспортный слой (CAN Gateway)
*   Абстракция над аппаратным интерфейсом (USB-CAN, Ethernet-CAN).
*   Обеспечивает прием/передачу сырых 29-битных фреймов.

### 2.2. Слой транзакций (Transaction Manager)
*   Управляет жизненным циклом команд согласно протоколу DDS-240.
*   Отслеживает последовательность: **COMMAND -> ACK -> [DATA] -> DONE**.
*   Считает `DONE` терминальным кадром транзакции: DATA после DONE является ошибкой порядка ответа Executor.
*   Реализует механизмы тайм-аутов (ожидание ответа от узла).
*   Обеспечивает потокобезопасность при обращении к разным узлам шины.

### 2.3. Слой абстракции устройств (Device Abstraction Layer - DAL)
*   **Универсальный узел (Base Node):** Реализует общие функции (0xF0xx), доступные на любой плате (Reboot, Info, Flash Commit).
*   **Специфические плагины (Device Plugins):** Расширяют базовый узел командами конкретного типа (Motion, Thermo, Fluidic).
*   ПО автоматически подгружает нужный плагин после идентификации типа устройства.

## 3. Механизм обнаружения и топология (Discovery)

Система строится на принципе динамического обнаружения узлов:
1.  **Bus Scan:** Сервисное ПО опрашивает диапазон адресов (0x02 - 0x7F) универсальной командой `SRV_GET_DEVICE_INFO (0xF001)`.
2.  **Identification:** Каждый узел возвращает структуру, содержащую:
    *   `Device Type ID` (например: `0x02` - Thermo, `0x01` - Motion, `0x03` - Fluidics).
    *   `Firmware Version`.
    *   `Unique Serial Number` (из MCU UID).
3.  **Topology Map:** ПО строит дерево устройств, доступных для настройки.

## 4. Единое командное пространство (0xFxxx Service Range)

Использование 16-битных кодов команд в сервисном диапазоне `0xFxxx` позволяет избежать конфликтов между разными типами оборудования:

| Диапазон | Группа | Описание |
| :--- | :--- | :--- |
| **0xF0xx** | **Universal** | Общие для всех: перезагрузка, инфо, сохранение во Flash. |
| **0xF1xx** | **Thermo** | Специфика DS18B20: скан шины, маппинг ROM ID. |
| **0xF2xx** | **Motion** | Специфика ШД: токи, лимиты, калибровка "дома". |
| **0xF3xx** | **Fluidics** | Специфика насосов: калибровка объема, тесты герметичности. |

## 5. Концепция "Warm Finger" (Интерактивная инсталляция)

Для минимизации ошибок при сборке и обслуживании, ПО поддерживает интерактивный маппинг:
1.  **Discovery:** ПО получает список всех физических датчиков/акторов на плате.
2.  **Stimulation:** Инженер физически воздействует на компонент (нагревает датчик пальцем, нажимает концевик, прокручивает мотор).
3.  **Feedback:** ПО визуально выделяет компонент, на котором изменились данные.
4.  **Binding:** Инженер назначает этому компоненту логическую роль в системе (например, "Датчик инкубатора 1") и фиксирует это командой `SET_CHANNEL_MAP`.

## 6. Преимущества подхода
*   **Универсальность:** Один сервисный кабель и одна программа для всего анализатора.
*   **Масштабируемость:** Добавление новой платы требует только создания нового плагина в ПО, не затрагивая архитектуру связи.
*   **Безопасность:** Сервисные команды отделены от операционных, что предотвращает случайное изменение настроек в рабочем цикле.

## 7. Модель доступа и безопасности (Hardware Access & Security)

### 7.1. Аппаратная фильтрация и адресная изоляция
Для обеспечения детерминированности системы и исключения коллизий применяется жесткая фильтрация трафика:
*   **Hardware Filtering:** Каждый исполнитель (Executor) настраивает аппаратные фильтры CAN-контроллера (bxCAN) на свой уникальный `Destination Address`. Сообщения, адресованные другим узлам, отсекаются на уровне кремния, не нагружая CPU.
*   **Broadcast (0x00):** Допускается только для системных команд (например, `EMERGENCY_STOP`), которые должны быть обработаны всеми узлами одновременно.

### 7.2. Объектная модель данных (Object Dictionary)
Доступ к параметрам железа унифицирован через структуру "Индекс/Суб-индекс" (по аналогии с CANopen):
*   **Индекс (16 бит):** Группирует объекты по функционалу (0xF1xx — термометрия, 0xF2xx — моторы).
*   **Суб-индекс (8 бит):** Указывает на конкретный экземпляр (датчик №3) или параметр (коэффициент калибровки).
*   Это позволяет ПО верхнего уровня работать с абстрактными "объектами устройства", упрощая автоматизацию сервисных процедур.

### 7.3. Транзакционная надежность (The Handshake)
Сервисный слой гарантирует доставку и исполнение через жизненный цикл транзакции:
1.  **Request (REQ):** ПО верхнего уровня инициирует запрос.
2.  **Acknowledge (ACK):** Узел обязан подтвердить получение и валидность команды в течение фиксированного тайм-аута (<100 мс). Если ACK не получен — узел считается `Offline`.
3.  **Completion (DONE/DATA):** Возврат результата или подтверждение завершения атомарного действия Executor.
4.  **Error Handling (NACK):** Если параметры невалидны или состояние узла не позволяет выполнить команду, возвращается код ошибки.

Для многокадровых ответов порядок должен быть детерминированным: `ACK -> DATA... -> DONE`. На STM32 bxCAN Executor это требует включенного `TransmitFifoPriority`. Если команда получила ACK, но DONE не пришел, а диагностический статус не показывает переполнений очередей или CAN faults, это трактуется как внутренний fault исполнителя: не создана или не работает доменная RTOS-задача, очередь или safety timer.

`DONE` означает завершение команды по ее контракту. Если физическая операция может быть локально измерена Executor-ом, ее длительность должна принадлежать Executor-у: например, `PUMP_RUN_DURATION` завершается `DONE` только после выключения насоса по истечении `duration_ms`. Дирижер ведет recipe sequencing, operation timeout, resource lock и технологические паузы, но не должен использовать `WAIT_MS` как основной механизм дозирования насосом.

### 7.4. Защита критических операций (Magic Keys)
Для предотвращения случайного повреждения системы опасные команды требуют передачи **защитного ключа (Magic Key)** в поле данных:
*   `SRV_REBOOT`: требует `0xDEAD`.
*   `SRV_FACTORY_RESET`: требует `0xFACE`.
*   `SRV_FLASH_COMMIT`: требует совпадения контрольной суммы RAM-конфигурации.

### 7.5. Валидация состояния (State Guard)
Исполнитель на программном уровне блокирует сервисный доступ, если он противоречит безопасности:
*   Запрет маппинга или калибровки во время выполнения рабочего цикла анализатора.
*   Валидация диапазонов (например, блокировка установки температуры выше физического предела безопасности).

## 8. Идентификация и типизация узлов (Identity & UID Management)

Для однозначного распознавания роли платы в системе используется трехуровневая модель идентификации:

### 8.1. Заводской идентификатор (MCU UID) — 96 бит
*   **Источник:** Уникальный серийный номер чипа STM32 (Unique Device ID), прошитый на заводе.
*   **Назначение:** "Отпечаток пальца" конкретного экземпляра железа. Позволяет отличить две идентичные платы друг от друга, даже если у них совпадают сетевые адреса (коллизия).

### 8.2. Тип устройства (Device Type ID) — 8 бит
*   **Источник:** Жестко прописывается в коде прошивки (Hardcoded).
*   **Назначение:** Определяет функциональную роль платы на шине.
    *   `0x01` — Плата управления шаговыми двигателями (Motion Board).
    *   `0x02` — Плата термометрии (Thermo Board).
    *   `0x03` — Плата управления жидкостной системой (Fluidic Board).
*   ПО верхнего уровня использует этот ID для выбора соответствующего плагина управления.
*   **Важно:** `Device Type ID` не равен сетевому `Node ID`. Например, Fluidics имеет `NodeID=0x30`, но `Device Type ID=0x03`.

### 8.3. Логический сетевой адрес (Node ID / Performer ID) — 8 бит
*   **Источник:** Хранится в `AppConfig` (Flash).
*   **Назначение:** Адрес для операционного обмена в рабочем цикле. Может быть изменен сервисным ПО для устранения коллизий или переназначения ролей.

### 8.4. Структура ответа на запрос идентификации (SRV_GET_DEVICE_INFO)
При получении команды `0xF001` узел обязан вернуть `ACK -> DATA... -> DONE`.

Первый DATA-фрейм должен начинаться одинаково для всех исполнителей:
1.  **Sub-Type:** `0x02` (DATA).
2.  **Sequence Info:** `0x80` для одиночного фрагмента или актуальный sequence byte для многофрагментного ответа.
3.  **Device Type ID:** роль платы (`0x01` Motion, `0x02` Thermo, `0x03` Fluidics).
4.  **FW Version:** 2 байта, `Major.Minor`.
5.  **Channel Count:** количество активных каналов/ресурсов.
6.  **Unique UID:** заводской номер чипа, передается в оставшихся байтах первого DATA и дополнительных DATA-фреймах.
