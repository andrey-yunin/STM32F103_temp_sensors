# Отчет о разработке документации CAN Протокола DDS 240

## Дата: Понедельник, 9 февраля 2026 г.

Этот отчет описывает процесс разработки и структуру документации для CAN Протокола, предназначенного для низкоуровневой связи между Дирижером (STM32H723) и Исполнителями (STM32F103) биохимического анализатора DDS 240.

---

## 1. Цель и структура документации

Целью данного этапа было создание всеобъемлющей и четкой документации, которая детально описывает CAN Протокол от физического уровня до прикладного, с учетом необходимости максимальной эффективности использования шины и совместимости с аппаратными ограничениями Исполнителей (STM32F103).

Документация была организована в отдельную директорию `readme/Commands_API/CAN_Protocol/` для лучшей структурированности и разделения высокоуровневых пользовательских команд от низкоуровневых протоколов.

## 2. Ключевые аспекты CAN Протокола

### 2.1. Физический и Канальный Уровень (`1_Physical_Layer.md`)

*   **Стандарт CAN**: Выбран **Классический CAN** (не CAN FD) для обеспечения совместимости с микроконтроллерами STM32F103, используемыми в Исполнителях.
*   **Скорость передачи**: 1 Мбит/с.
*   **Длина данных (DLC)**: Максимально 8 байт для всех кадров, что соответствует ограничениям Классического CAN.
*   **Терминация**: Стандартная 120 Ом.
*   **Распиновка**: Определена для DB9 Male разъема.

### 2.2. Формат Кадра (`2_Frame_Format.md`)

*   **CAN ID (29-бит Extended)**: Структурирован для кодирования приоритета, типа сообщения (COMMAND, ACK, NACK, DATA/DONE/LOG), адреса назначения и адреса источника.
*   **Типы Сообщений**: Четко определены битовые поля для COMMAND, ACK, NACK, и объединенного типа для DATA/DONE/LOG, различаемых по Sub-Type.
*   **Адреса Узлов**: Предложена схема адресации для Хоста, Дирижера и различных Исполнителей.
*   **Переменная DLC**: Длина полезной нагрузки является переменной и зависит от типа сообщения, что обеспечивает эффективное использование пропускной способности шины.

### 2.3. Прикладной Уровень (`3_Application_Layer.md`)

*   **Последовательность Выполнения Команд**: Детально описан типовой цикл обмена COMMAND -> ACK -> DATA (опционально) -> DONE, включая обработку ошибок (NACK).
*   **Структуры Sub-Payload**: Определены детальные структуры для `DONE`, `DATA` (включая информацию о последовательности для многокадровых передач) и `LOG` (включая уровни логирования и формат сообщения).

### 2.4. Примеры (`4_Examples.md`)

*   Предоставлены конкретные пошаговые примеры низкоуровневой CAN-коммуникации между Дирижером и Исполнителем, иллюстрирующие передачу команды `MOTOR_ROTATE`, включая отправку COMMAND, ACK, и DONE кадров с учетом определенных DLC.

### 2.5. Набор Низкоуровневых Команд (`5_Low_Level_Commands.md`)

*   Инициирован список низкоуровневых команд, которые исполнители понимают и выполняют (например, `MOTOR_ROTATE`, `PUMP_RUN_DURATION`), с указанием их кодов и параметров.

### 2.6. Стратегия Упаковки Параметров (`6_Parameter_Packing.md`)

*   Обоснована и задокументирована философия эффективной упаковки параметров, ориентированная на минимизацию использования байтов при сохранении ясности и выравнивания для удобства обработки на Исполнителях.
*   Проведен анализ и обоснование структуры полезной нагрузки для примеров команд `MOTOR_ROTATE` и `PUMP_RUN_DURATION`.

## 3. Заключение

Разработанная документация служит прочной основой для дальнейшей реализации CAN-коммуникации в проекте DDS 240, обеспечивая ясность, эффективность и соответствие аппаратным ограничениям.
